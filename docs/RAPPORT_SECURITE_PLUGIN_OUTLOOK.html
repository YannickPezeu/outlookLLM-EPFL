1. Resume du projet

EPFL Mail AI est un add-in Microsoft Outlook qui assiste le personnel dirigeant EPFL dans la preparation de reunions. Il analyse les emails echanges avec les participants d'une reunion et genere un briefing structure via l'API LLM interne RCP EPFL.

2. Architecture

Point cle : L'application est 100% client-side (navigateur). Il n'y a aucun backend applicatif. Les fichiers statiques sont servis par un conteneur nginx sur le cluster Kubernetes IC.
+-------------------+ +-------------------+ +----------------------+ | Outlook Web | | expert-finder. | | inference.rcp. | | (navigateur) |----->| epfl.ch (k8s) |----->| epfl.ch | | | | nginx static + | | API LLM interne | | Add-in taskpane | | CORS proxy | | EPFL | +-------------------+ +-------------------+ +----------------------+ | | HTTPS (tokens delegues) v +-------------------+ | graph.microsoft. | | com | | Microsoft Graph | | API v1.0 | +-------------------+

Flux de donnees

  1. L'utilisateur ouvre l'add-in dans Outlook Web
  2. L'add-in s'authentifie via MSAL.js (OAuth 2.0 PKCE) aupres d'Entra ID
  3. L'add-in interroge Microsoft Graph API avec le token delegue pour lire les emails/calendrier
  4. Les emails pertinents sont envoyes a l'API RCP EPFL pour analyse LLM
  5. Le resultat (briefing) est affiche dans le taskpane

3. Securite - Garanties

Lecture seule. L'add-in ne modifie, supprime, envoie ou deplace jamais aucun email ou evenement calendrier.
GarantieDetail
Lecture seulePermissions Graph API : Mail.Read, Calendars.Read, User.Read uniquement. Aucune permission d'ecriture.
Pas de backendAucun serveur applicatif. Le code tourne integralement dans le navigateur de l'utilisateur. Le serveur nginx ne sert que des fichiers statiques et un proxy CORS.
Donnees internesLes contenus emails sont envoyes uniquement a l'API RCP EPFL (inference.rcp.epfl.ch), hebergee sur l'infrastructure EPFL. Aucune donnee ne quitte le perimetre EPFL.
Pas de stockageAucune donnee email n'est persistee. Les donnees existent uniquement en memoire pendant la session du navigateur. Le localStorage ne stocke que les preferences utilisateur (cle API, modele).
Permissions delegueesToutes les permissions sont Delegated (pas Application). Chaque utilisateur n'accede qu'a sa propre boite mail via son propre token OAuth.
HTTPS partoutToutes les communications sont chiffrees (HTTPS/TLS).

Role du serveur nginx

Le serveur nginx a deux fonctions :

  1. Servir les fichiers statiques (HTML, JS, CSS) de l'add-in
  2. Proxy CORS pour l'API RCP : relayer les requetes du navigateur vers inference.rcp.epfl.ch en ajoutant les en-tetes CORS necessaires. Ce proxy ne stocke, ne modifie et ne log aucune donnee.

8. Surface d'attaque et mitigations

RisqueMitigation
Vol de token OAuthFlux PKCE (pas de client secret expose). Tokens de courte duree. Stockage en memoire/sessionStorage (pas localStorage).
Cle API RCP en localStorageRisque accepte pour l'UX. La cle est personnelle et revocable. Alternative future : stockage dans un vault.
XSS dans le contenu emailLe contenu email est affiche via React (echappement automatique). Le rendu Markdown des reponses LLM est sanitise avec DOMPurify avant injection HTML.
Injection dans les prompts LLMRisque inherent aux LLM. Impact limite : l'add-in est en lecture seule, un prompt malveillant ne peut pas declencher d'action destructive.
Acces non autorise aux emailsPermissions deleguees uniquement. Chaque utilisateur n'accede qu'a ses propres emails via son token.
Donnees en transitHTTPS/TLS pour toutes les communications.
